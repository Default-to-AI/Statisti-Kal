Problem

Six math functions are copy-pasted identically across two calculators; two more exist only in HypothesisTestingCalculator. Any bug fix must happen twice. No shared seam for tests.

Solution

Extract all CDF/inverse/PDF functions into src/lib/math/distributions.ts. Both calculators import from one place.

Wins

  • Locality: math bugs concentrate in one module
  • Leverage: one interface, 2+ importers, N tests
  • Interface shrinks: each calculator drops ~180 LOC
  • Test surface: pure functions, trivially testable
  • Delete ~6 duplicate functions (~200 lines)

usePowerAnalysis(params) → result — hook, ~120 LOC
HypothesisTestingCalculator — UI composition only, ~600 LOC

Problem

315 KB file contains math, types, helper components, calculation engines, decision logic, CI logic, power analysis, and 2300 lines of JSX — all with no seam between them. The interface is the entire file; the implementation is the entire file. Depth ≈ 0.

Solution

Extract calculation logic into three hooks (useHypothesisTest, useConfidenceInterval, usePowerAnalysis). Each hook has a small interface (params in, result out) and a deep implementation. The remaining component composes UI from primitives.

Wins

  • Locality: test logic bugs concentrate in hooks, not scattered across 3421 lines
  • Leverage: hooks testable without rendering; UI testable without re-computing
  • Interface shrinks: each hook has ~5 params, ~10 result fields vs. 50+ interleaved state vars
  • Bundle target: < 50KB (from 315KB)

6. Deepen: FormulaSheet — the undocumented third monolith

Speculative in-process

After — data-driven, searchable

Problem

103 KB of static KaTeX markup with no search, no filtering, no context awareness. Students must scroll through everything to find one formula. STRATEGY.md calls for "Context-Aware Lookup" to replace this "monolithic dump."

Solution

Separate formula data from presentation. Create a structured formulas.ts data file with categories, tags, and LaTeX strings. Build a search/filter UI on top.

Wins

  • Locality: formula content edits don't touch UI code
  • Leverage: search + context-aware inline blocks serve multiple flows
  • Matches STRATEGY.md track: "Formula Sheet → Context-Aware Lookup"

Top Recommendation

Start with Candidate 1: Extract the Math module.

It's the only candidate with zero risk — pure functions, no UI, no state, no side effects. It unblocks Candidates 3 and 5 (both monolith splits depend on extracted math). It creates the first testable seam in the codebase (currently 0 tests, 0% coverage). And it deletes ~200 lines of exact duplicates on day one.

After math extraction, proceed to Candidate 4: Wire signature elements — it's quick (class additions only, no refactoring) and makes the DESIGN.md investment visible immediately.
